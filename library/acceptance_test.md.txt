# Curie Trust - Acceptance Test Plan v1.2

## 1. Introduction

This document outlines the acceptance criteria and testing strategy for each phase of the Curie Trust Financial Control Center project, as defined in the revised Project Requirements Specification (PRS), Section 10. The plan follows a user-feature-centric approach, verifying that each vertical slice of functionality is complete and correct before proceeding.

---

## 2. Phase 0-4: Core Functionality (Completed)

All acceptance criteria for phases 0 through 4 and 6.1 (Core Feature Verification) have been met and are considered stable.

---

## 3. Phase 5: Automated Market Data & Layered Returns

**Objective**: Automate portfolio pricing using a multi-provider strategy and introduce advanced return metrics.

**Result**: CURRENT PHASE

### 5.1 Multi-Provider Pricing Service

**Objective**: Verify that the backend polling service correctly uses different API providers based on the asset type of a holding.

**Acceptance Criteria**:
- [ ] **Setup**: 
    1. Ensure `.env` file contains valid keys for `MASSIVE_API_KEY` and `ALPHA_VANTAGE_API_KEY`.
    2. Purge existing `holdings` data.
    3. Import a new holdings CSV containing at least one `Common Stock` (e.g., AAPL) and one `Mutual Fund Open` (e.g., VTSAX). Ensure the `asset_type` column is correctly populated.
- [ ] **Execution**: Manually trigger the market data refresh for all holdings by running the script `./scripts/refresh_eod.sh` or calling the `/api/market-data/refresh-eod` endpoint.
- [ ] **Verification (Logs)**: 
    1. Check the backend server logs.
    2. Confirm a log entry shows the polling service processing `AAPL` via `massive_provider`.
    3. Confirm a log entry shows the polling service processing `VTSAX` via `alphavantage_provider`.
- [ ] **Verification (Database)**: 
    1. Query the `holdings` table in `trust.db`.
    2. Both the AAPL and VTSAX rows should have a non-null `market_value` and a recent `last_price_timestamp`.
    3. The `last_price_update_failed` flag for both should be `0` (or `False`).
- [ ] **Verification (UI)**: 
    1. Navigate to the Portfolio screen.
    2. The "Last Updated" column for both AAPL and VTSAX should display the recent timestamp without any error styling.

### 5.2 Layered Returns & Tax Facts

**Objective**: Verify the end-to-end functionality of entering tax data and seeing it reflected in after-tax return calculations.

**Acceptance Criteria**:
- [ ] **UI Implementation**: The `TaxFactsEditor` component is present and functional on the Data & Settings screen.
- [ ] **Data Persistence (Create/Update)**: 
    1. Navigate to Data & Settings -> Personal Inputs.
    2. Select a tax year (e.g., 2024) from the dropdown.
    3. Enter valid numbers in all four tax data fields (Federal/State Taxable Income, Federal/State Total Tax).
    4. Click "Save 2024 Facts".
    5. Verify a success message is shown.
- [ ] **Data Persistence (Read)**:
    1. Select a different tax year, then select the original test year (e.g., 2024) again.
    2. Verify the previously entered numbers are correctly re-loaded into the form fields.
- [ ] **Backend Verification**:
    1. Using a SQLite browser, query the `tax_year_facts` table in `data/trust.db`.
    2. Confirm that a row exists for the test year (e.g., 2024) and that all columns contain the correct data saved from the UI.
- [ ] **After-Tax Calculation Display**:
    1. Navigate to the Portfolio screen.
    2. Verify that a metric for "After-Tax Return" is displayed alongside Gross and After-Fee returns.
    3. Verify the Returns Sankey correctly visualizes the `Gross Return -> Taxes -> After-Tax Return` flow, using the entered tax data to perform its calculations.
