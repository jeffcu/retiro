# Curie Trust Financial Control Center
## Master Design Specification (MDS) v1.0

---

### 1. Overview

This document outlines the technical architecture for the Curie Trust Financial Control Center, a local-first desktop application designed for comprehensive financial tracking, analysis, and forecasting. The system ingests data from various financial institutions, normalizes it against an internal model, enriches it with market data via external APIs, and presents insights through a Sankey-forward user interface.

The prime directive is to answer three core questions with high clarity:
1.  What is our latest net worth?
2.  Where did our income go?
3.  What is our investment rate of return (gross, after fees, after taxes)?

### 2. System Architecture

The application will follow a simple, robust, local-first architecture.

*   **Backend:** A Python application responsible for all business logic, data processing, storage, and external API communication. This ensures deterministic calculations and direct control over the data pipeline.
*   **Frontend:** A web-based user interface (e.g., built with React or Vue) served locally by the Python backend. This allows for a modern, interactive UX without the complexities of a native GUI framework.
*   **Storage:** A single-file SQLite database for all persistent data. This provides transactional integrity and simplifies backup/restore operations.
*   **Communication:** The frontend will communicate with the backend via a local REST or WebSocket API.

```mermaid
graph TD
    A[User] --> B{Frontend (Web UI)};
    B <--> C{Backend API (Python/FastAPI)};
    C --> D[Business Logic];
    D <--> E[SQLite Database];
    D <--> F(External API: Alpha Vantage);
    G[CSV/Spreadsheet Files] --> C;
```

### 3. Core Modules

The system is decomposed into the following distinct modules within the `src` directory:

*   `src/data_model.py`: Defines the core Python data structures (`Transaction`, `Account`, `Holding`, etc.) using dataclasses and Enums.
*   `src/database.py`: Manages all interaction with the SQLite database, including schema creation, CRUD operations, and transaction management.
*   `src/importers/`: A package for data ingestion. Contains parsers for different CSV formats and the logic for mapping, normalization, and deduplication.
*   `src/rules_engine.py`: Handles the logic for automatically categorizing transactions based on user-defined rules (e.g., merchant regex matching).
*   `src/market_data/`: A package for fetching external market data. Contains the provider interface and a specific implementation for Alpha Vantage.
*   `src/analysis.py`: Contains functions for financial calculations, generating Sankey diagram data, calculating portfolio returns, and running forecasts.
*   `src/main.py` or `src/api.py`: The main application entry point. Runs the web server (e.g., FastAPI) and exposes the API endpoints for the frontend.

### 4. Data Model & Storage

*   **Database:** SQLite
*   **Location:** `data/trust.db` (within the project structure, but outside `src`)
*   **Schema:** The database schema will directly mirror the entities defined in `src/data_model.py` and the PRS Section 8.

**Key Tables:**
*   `transactions`: Stores all financial events (income, expense, transfer, capex).
*   `accounts`: Represents financial accounts (e.g., checking, credit card).
*   `assets`: Represents major assets, primarily for tracking basis (e.g., 'Main Residence').
*   `holdings`: Stores portfolio positions (e.g., 100 shares of AAPL in Account X).
*   `price_history`: Caches historical and intraday price quotes for securities.
*   `rules`: Stores user-defined rules for transaction categorization.
*   `import_profiles`: Saves column mappings and settings for specific CSV formats.

### 5. API Interfaces

#### 5.1 Internal Backend API

The backend will expose RESTful endpoints for the frontend, such as:
*   `GET /api/net-worth`: Returns the latest calculated net worth.
*   `GET /api/sankey/income?period=YTD`: Returns data formatted for the Income->Uses Sankey.
*   `POST /api/import/csv`: Accepts a CSV file for ingestion.
*   `GET /api/transactions?category=Travel`: Retrieves transactions based on filters.
*   `PUT /api/transaction/{id}`: Updates a single transaction (e.g., manual re-categorization).

#### 5.2 External API (Alpha Vantage)

*   **Provider:** Alpha Vantage
*   **Authentication:** API Key (`ALPHA_VANTAGE_API_KEY`) must be used for all requests.
*   **Module:** `src/market_data/alpha_vantage_provider.py`

**Required API Functions:**
1.  `GLOBAL_QUOTE`: For fetching the latest price for a single symbol. Ideal for the hourly refresh of the Top 25 holdings.
2.  `TIME_SERIES_DAILY_ADJUSTED`: For backfilling historical prices and for the after-market EOD refresh.

**Implementation Details:**
*   The provider will implement a `get_quotes(symbols: list)` method.
*   To respect rate limits (5 calls per minute / 100 per day on free tier), the implementation must batch requests and manage timing carefully.
*   The polling algorithm from PRS Section 6.3 will be implemented in a dedicated service that uses this provider.
*   All API responses will be cached in the `price_history` table to minimize redundant calls.

### 6. Security

As a local-first application, the primary security concerns are key management and data integrity.

*   **API Key Storage:** The `ALPHA_VANTAGE_API_KEY` must not be hardcoded. It will be stored in a configuration file (e.g., `.env`) that is explicitly excluded from version control via `.gitignore`. The application will load the key into its environment at runtime.
*   **Data Storage:** The `trust.db` SQLite file contains sensitive financial data. While encryption-at-rest is optional for v1.0, the design should accommodate a future implementation using a library like `sqlcipher`.
*   **Input Validation:** All data from imported files will be strictly validated and sanitized before being inserted into the database to prevent corruption or injection attacks.
*   **No Inbound Network Access:** The application server will bind to `localhost` by default, ensuring it is not exposed to the local network.

### 7. Phased Implementation Plan

Development will follow the phased plan outlined in the PRS (Section 10) to ensure iterative delivery of value.

*   **Phase 0:** Transactions import + Sankey Income→Categories (basic)
*   **Phase 1:** Rules engine + tags + drilldowns
*   **Phase 2:** CapEx + Real Estate basis + Estate Flow Sankey
*   **Phase 3:** Holdings snapshots + portfolio screens + quote cache
*   **Phase 4:** Scheduled drip pricing (top 25 hourly, long tail EOD)
*   **Phase 5:** Fees/taxes entry + layered returns + returns Sankey
*   **Phase 6:** Forecast scenarios + planned CapEx events

Curie Trust Financial Control Center — Project Requirements Specification (PRS) v1.0

1) Mission

Build a local-first financial control center for the Curie Trust that continuously answers three questions with high clarity:
	1.	What is our latest net worth (right now)?
	2.	Where did our income go (by top budget category and projects/tags)?
	3.	What rate of return are we getting on our investments (before fees, after fees, after taxes)?

The system ingests periodic CSV/spreadsheet uploads from multiple financial institutions, normalizes them into Curie Trust’s internal data model, and keeps market values fresh via API “drip” updates throughout the day, with hourly polling for the top 25 holdings and after-market polling for the long tail (250+).

⸻

2) Core UX Requirements (Sankey-forward)

Sankey diagrams are not decorative. They are the primary language for “money as flow / conversion / leakage.”

2.1 Required Sankey Diagrams (Must-Haves)

A) Income → Uses of Money (Home Screen Primary Sankey)
	•	Left node: Total Income for selected period
	•	Middle split: Expenses (consumption), CapEx (asset-building), Net Savings/Surplus
	•	Right split for Expenses: Top 5–7 budget categories, remainder as Other
	•	CapEx flows to specific asset(s) (e.g., “Main Residence Basis”)

B) Estate Flow (Net Worth / Estate Screen Sankey)
	•	Left: Income + portfolio distributions (if modeled)
	•	Flows to:
	•	Consumption Expenses
	•	CapEx → Real Estate Basis
	•	Portfolio (invested) → Market Return → Fees → Taxes → After-Tax Return
	•	This diagram must clearly show “conversion” (CapEx) vs “leakage” (fees/taxes).

C) Portfolio Return Waterfall as Sankey (Portfolio Screen)
	•	Left: Gross Market Return
	•	Splits to: Fees, Taxes, After-Tax Return
	•	Always displayed with matching numeric metrics (%, $).

2.2 Sankey Interaction Rules
	•	Node/flow click drills down to:
	•	Categories → merchants → transactions
	•	CapEx → asset ledger entries
	•	Fees/taxes → supporting manual records
	•	Must support: month, YTD, rolling 12 months, custom range.
	•	Must cap complexity: max ~9–12 nodes per Sankey; enforce “Other” aggregation.

⸻

3) Financial Semantics (Non-Negotiable)

3.1 Transaction Cashflow Types (Mutually Exclusive)

Every transaction is exactly one:
	•	Income
	•	Expense (consumption)
	•	Transfer (between trust-controlled accounts; net-zero)
	•	CapEx (capital improvement; increases asset basis)
	•	(Optional future) Investment transaction

Expenses drive “spend/burn.”
CapEx is excluded from “spend/burn” but included in net worth via asset basis.

3.2 Tags (Non-Exclusive, Multi-Label)
	•	A transaction can have 0..N tags (e.g., Travel AND Italy26).
	•	Tag subtotals are non-exclusive by default (overlap is expected).

3.3 Real Estate CapEx Accounting
	•	Trust has no mortgage; liabilities default $0 unless user adds any.
	•	CapEx transactions link to an Asset and increase asset basis.
	•	Market value is tracked separately via snapshots (manual or appraised).

⸻

4) Screens (UI Requirements Summary)

4.1 Home (Command Center)

Priority order:
	1.	Latest Net Worth Hero
	2.	Income→Uses Sankey
	3.	Portfolio Return Sankey (Gross→Fees→Taxes→After-Tax)
	4.	Alerts/insights (spend spikes, missing imports, stale prices)

4.2 Cashflow
	•	Monthly income vs expenses trend
	•	Category drilldowns
	•	Toggle include/exclude CapEx (default exclude)

4.3 Projects/Tags
	•	Tag selector (single or multi)
	•	Tag-specific totals, trends, and “where it went” Sankey filtered by tag

4.4 Real Estate / Estate
	•	Asset basis ledger
	•	CapEx over time
	•	Market value snapshots
	•	Estate Flow Sankey

4.5 Portfolio
	•	Value over time (snapshots + intraday updates)
	•	Allocation and concentration
	•	Return layer metrics:
	•	Before fees
	•	After fees
	•	After taxes
	•	Return Sankey always visible here

4.6 Forecast
	•	Scenario-based forecast:
	•	Expenses (consumption)
	•	Income
	•	Expected returns
	•	One-time planned CapEx events
	•	Outputs: projected portfolio, net worth, runway

4.7 Data & Settings
	•	CSV import, mapping profiles, dedupe logs
	•	Rules editor (merchant→category/tags/type)
	•	Manual entry: fees, taxes, market value snapshots
	•	Backup/restore

⸻

5) Data Ingestion Requirements (CSV/Spreadsheet Flexibility)

5.1 Supported Inputs (v1.0)
	•	Bank transactions CSV
	•	Credit card transactions CSV
	•	Brokerage/retirement holdings snapshots spreadsheet/CSV
	•	Optional: account balance snapshots

5.2 Import Pipeline (Must)
	1.	Upload file(s) (drag/drop)
	2.	Detect type via column heuristics
	3.	Preview + mapping UI (first time per institution)
	4.	Save Import Profile (mapping + sign rules + date formats)
	5.	Normalize into internal schema
	6.	Deduplicate (hash-based fallback)
	7.	Produce import audit log + “what changed” summary

5.3 Categorization & Type Assignment
	•	Rules engine:
	•	merchant/description regex → cashflow_type, category, tags
	•	transfer detection rules (pair matching or known transfer descriptors)
	•	Manual override always available; edits can generate “rule suggestions.”

⸻

6) Market Value Drip Updates (API-driven pricing)

6.1 Requirements
	•	Maintain a Price Cache and update holdings market values throughout the day.
	•	Polling schedule:
	•	Top 25 holdings: hourly during market hours
	•	Long tail 250+: after-market (end-of-day) polling
	•	Must support multiple asset types where possible (stocks/ETFs; others optional).
	•	Must degrade gracefully if API limits are hit (backoff, partial refresh, stale flags).

6.2 Vendor Strategy (Pluggable Provider Interface)

Implement a provider interface so Curie Trust can switch data sources without rewriting the app:

Provider Interface (example)
	•	get_quotes(symbols[]) -> {symbol: {price, timestamp, source}}
	•	get_eod_prices(symbols[], date) -> ...
	•	get_metadata(symbol) -> {name, currency, type}

Reality check on rate limits: many “free” tiers are too limited for frequent polling; design must include caching and batching. Alpha Vantage explicitly documents free-tier usage constraints and offers premium scaling.  
If you consider alternatives, Marketstack also publishes very low free quotas (e.g., limited monthly requests), which reinforces the need for batching and tiering.  

6.3 Polling Algorithm
	•	Rank holdings by current market value daily.
	•	Select top 25 by value for “frequent set.”
	•	During market hours:
	•	Once per hour: batch request for top 25 (or smaller batches per provider limits)
	•	After market close:
	•	EOD refresh for all symbols not refreshed within the day
	•	Use:
	•	exponential backoff on errors
	•	circuit breaker if provider fails repeatedly
	•	stale-data indicators in UI (e.g., “prices last updated 2h ago”)

6.4 Storage & Audit
	•	Store every quote with timestamp (or store latest + optional history).
	•	Keep provenance: provider name + request time.
	•	Log refresh runs: number updated, failures, duration.

⸻

7) Returns Before Fees / After Fees / After Taxes

7.1 Inputs
	•	Portfolio value time series:
	•	From holdings snapshots + intraday quotes
	•	User-entered Fee Records
	•	User-entered Tax Records from tax returns (annual totals and/or allocated)

7.2 Output Metrics (Required)

For selected period:
	•	Gross Return ($, %): value change adjusted for contributions/withdrawals if available; otherwise snapshot-to-snapshot change labeled “approx.”
	•	After Fees Return: gross minus fees in period
	•	After Taxes Return: after-fees minus taxes in period

All return views must label:
	•	“Exact” vs “Estimated” based on data completeness.

⸻

8) Internal Data Model (Minimum Entities)
	•	Transaction
	•	Account
	•	Institution
	•	Category
	•	Tag + TransactionTag
	•	Asset (Real Estate, Portfolio, etc.)
	•	AssetValueSnapshot (manual market value)
	•	CapExEntry (or CapEx as Transaction with asset_id)
	•	HoldingSnapshot (positions)
	•	PriceQuote (intraday/EOD)
	•	FeeRecord
	•	TaxRecord
	•	ImportProfile + ImportRun + ImportRowHash (dedupe/audit)

⸻

9) Non-Functional Requirements
	•	Local-first storage (SQLite recommended)
	•	Fast UI response (<1s for common filters on typical dataset sizes)
	•	Deterministic calculations (same inputs → same outputs)
	•	Safe importing (idempotent; repeat import doesn’t duplicate)
	•	Backups (single file export + restore)
	•	Security: encryption at rest optional v1.0; strongly recommended if later cloud sync exists

⸻

10) Phased Delivery (Thin Threads)

Phase 0: Transactions import + Sankey Income→Categories (basic)
Phase 1: Rules engine + tags + drilldowns
Phase 2: CapEx + Real Estate basis + Estate Flow Sankey
Phase 3: Holdings snapshots + portfolio screens + quote cache
Phase 4: Scheduled drip pricing (top 25 hourly, long tail EOD)
Phase 5: Fees/taxes entry + layered returns + returns Sankey
Phase 6: Forecast scenarios + planned CapEx events

⸻

11) Acceptance Criteria (Key)
	•	Home screen shows: Net Worth Hero + Income Sankey + Return Sankey with drilldowns.
	•	Importing the same CSV twice produces no duplicates.
	•	A transaction can be tagged with multiple tags and contributes to both tag totals.
	•	Marking a transaction as CapEx:
	•	removes it from “spending”
	•	increases the selected asset’s basis ledger
	•	Top 25 holdings refresh hourly during market hours; others refresh after market; UI clearly indicates quote staleness.
	•	Portfolio screen displays before fees / after fees / after taxes and the Sankey that explains the difference.

# Curie Trust - Acceptance Test Plan v1.0

## 1. Introduction

This document outlines the acceptance criteria and testing strategy for each phase of the Curie Trust Financial Control Center project, as defined in the Project Requirements Specification (PRS), Section 10. The purpose is to ensure that each development phase delivers the intended functionality and meets quality standards before proceeding to the next.

Each phase includes:
- **Objective**: A high-level goal for the phase.
- **Acceptance Criteria**: A checklist of testable outcomes required for completion.
- **Potential Risks & Failure Points**: Likely areas where issues may arise.
- **Mitigation & Verification Strategy**: How we will test for these issues and what to do if they are found.

---

## 2. Phase 0: Transaction Import & Basic Sankey

**Objective**: Establish the foundational data ingestion pipeline. The system must accept a standard transaction CSV, and the backend must be able to initialize the database and serve a placeholder API endpoint.

**Acceptance Criteria**:
- [x] The application starts without errors (`uvicorn src.main:app`).
- [x] On first run, a `data/trust.db` file is created.
- [x] Inspecting `trust.db` with an SQLite tool reveals a `transactions` table with the correct columns.
- [x] Sending a POST request to `/api/import/csv` with a sample CSV file returns a `200 OK` status and a JSON response indicating the number of parsed transactions.
- [x] Sending the same request with a non-CSV file returns a `400 Bad Request` error.
- [x] Sending a GET request to `/api/sankey/income` returns a `200 OK` status with valid placeholder JSON data.

**Potential Risks & Failure Points**:
1.  **Database Initialization Failure**: Permissions issues or SQL syntax errors could prevent the database file or table from being created.
2.  **CSV Parsing Logic is Brittle**: The initial parser may fail on common variations in CSVs (e.g., different line endings, encoding, header names).
3.  **API Endpoint Misconfiguration**: FastAPI routes may not be set up correctly, leading to `404 Not Found` errors.

**Mitigation & Verification Strategy**:
1.  **DB Verification**: Manually delete `trust.db` and restart the app to confirm it's recreated. Use a command-line tool (`sqlite3`) to run `PRAGMA table_info(transactions);` and verify the schema.
2.  **CSV Test Cases**: Create a small suite of test CSV files: one "perfect" file, one with different column orders, one with extra columns, and one with non-UTF-8 characters. The current test is manual; future phases may require automated unit tests for the importer.
3.  **API Endpoint Testing**: Use `curl` or a tool like Postman to directly test the API endpoints, verifying headers, status codes, and response bodies.

---

## 3. Phase 1: Rules Engine & Tagging

**Objective**: Implement the core logic for automatic transaction categorization and tagging based on user-defined rules.

**Acceptance Criteria**:
- [ ] A `rules` table exists in the database.
- [ ] API endpoints exist to `CREATE`, `READ`, `UPDATE`, and `DELETE` categorization rules (e.g., `POST /api/rules`).
- [ ] After creating a rule (e.g., description containing "Coffee Shop" maps to category "Food"), importing a CSV with matching transactions results in those transactions having `category = "Food"` in the database.
- [ ] A transaction can be associated with multiple tags.
- [ ] API drill-down endpoints work (e.g., `GET /api/transactions?category=Food`).

**Potential Risks & Failure Points**:
1.  **Regex Complexity**: Poorly written regex in rules could cause performance issues or incorrect matches.
2.  **Rule Application Order**: It may be unclear which rule takes precedence if a transaction matches multiple rules.
3.  **Data Integrity**: Updating a transaction's category/tags might fail or have unintended side effects.

**Mitigation & Verification Strategy**:
1.  **Rule Unit Tests**: The rules engine logic must be unit-tested in isolation with a variety of test descriptions and regex patterns.
2.  **Define Precedence**: The system must have a clear, documented, and testable rule for precedence (e.g., order of creation, a user-defined priority field).
3.  **Transactional Integrity**: All database modifications for a given import or update should be wrapped in a single transaction to ensure atomicity.

---

## 4. Subsequent Phases (High-Level Plan)

This section will be detailed as development approaches each phase.

### Phase 2: CapEx & Real Estate
- **Verification**: Manually flag a transaction as CapEx. Verify it's removed from expense reports and that the corresponding asset's basis increases in an `assets` table. Check the Estate Flow Sankey data.

### Phase 3: Holdings & Portfolio
- **Verification**: Import a holdings CSV. Verify positions are created correctly in a `holdings` table. Verify the `price_history` table is created. Test API endpoints for portfolio composition.

### Phase 4: Market Data Drip
- **Verification**: Configure the Alpha Vantage API key. Manually trigger the fetch process. Verify the `price_history` table is populated with fresh quotes. Monitor application logs for evidence of hourly/EOD scheduled jobs. Test graceful failure by providing an invalid API key and observing logged errors.

### Phase 5: Layered Returns
- **Verification**: Manually create `FeeRecord` and `TaxRecord` entries via an API. Using a test portfolio with known values, verify that the calculated Gross, After-Fees, and After-Taxes returns match calculations performed in a separate spreadsheet.

### Phase 6: Forecasting
- **Verification**: Create a simple forecast scenario (e.g., fixed income, fixed expenses, 0% return). Verify that the projected net worth matches a simple linear projection calculated manually. Add a one-time future event and verify the projection changes as expected.

from dataclasses import dataclass, field
from datetime import date
from decimal import Decimal
from enum import Enum


class CashflowType(Enum):
    """ PRS Section 3.1: Mutually exclusive cashflow types. """
    INCOME = "Income"
    EXPENSE = "Expense"
    TRANSFER = "Transfer"
    CAPEX = "Capital Expenditure"


@dataclass
class Transaction:
    """
    Represents a single, normalized financial transaction based on the internal data model.
    See PRS Section 8.
    """
    transaction_id: str  # A unique identifier, possibly a hash of the raw data.
    account_id: str
    transaction_date: date
    amount: Decimal
    description: str  # The original description from the financial institution.
    merchant: str | None = None  # Normalized merchant name.
    category: str | None = None
    cashflow_type: CashflowType | None = None
    tags: list[str] = field(default_factory=list)
    is_transfer: bool = False  # Derived from cashflow_type for quick filtering.

    # Link to other entities
    asset_id: str | None = None  # For CapEx transactions, linking to a specific asset.

    # Audit trail
    import_run_id: str | None = None
    raw_data_hash: str | None = None

import sqlite3
from pathlib import Path
from typing import List
from src.data_model import Transaction

# Per MDS, the database is a single file in the data/ directory.
# We resolve the path to its absolute form to avoid ambiguity.
DB_FILE = (Path(__file__).parent.parent / "data" / "trust.db").resolve()

def get_db_connection():
    """ Establishes a connection to the SQLite database. """
    # Ensure the data directory exists
    DB_FILE.parent.mkdir(exist_ok=True)
    
    # Use the string representation of the path for sqlite3.connect
    conn = sqlite3.connect(str(DB_FILE))
    conn.row_factory = sqlite3.Row
    return conn

def initialize_database():
    """
    Creates the necessary tables in the database if they don't already exist,
    based on the schema defined in the MDS and src/data_model.py.
    """
    print(f"--- Initializing database at: {DB_FILE} ---")
    conn = get_db_connection()
    cursor = conn.cursor()

    # Per Phase 0 requirements, create the 'transactions' table.
    cursor.execute("""
    CREATE TABLE IF NOT EXISTS transactions (
        transaction_id TEXT PRIMARY KEY,
        account_id TEXT NOT NULL,
        transaction_date TEXT NOT NULL,
        amount REAL NOT NULL,
        description TEXT NOT NULL,
        merchant TEXT,
        category TEXT,
        cashflow_type TEXT,
        is_transfer INTEGER,
        asset_id TEXT,
        import_run_id TEXT,
        raw_data_hash TEXT UNIQUE
    );
    """)

    conn.commit()
    conn.close()
    print("--- Database initialization complete. ---")

def save_transactions(transactions: List[Transaction]):
    """
    Saves a list of Transaction objects to the database.
    Uses a transaction to ensure all or nothing is saved.
    """
    if not transactions:
        return

    conn = get_db_connection()
    cursor = conn.cursor()
    
    # The UNIQUE constraint on raw_data_hash will prevent duplicates on re-import.
    sql = """INSERT OR IGNORE INTO transactions (
                 transaction_id, account_id, transaction_date, amount, description, 
                 merchant, category, cashflow_type, is_transfer, asset_id, 
                 import_run_id, raw_data_hash
             ) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?);"""
    
    # This data tuple must match the SQL columns exactly (12 items).
    data_to_insert = [
        (
            t.transaction_id, t.account_id, t.transaction_date.isoformat(), t.amount, t.description,
            t.merchant, t.category, t.cashflow_type.value if t.cashflow_type else None,
            1 if t.is_transfer else 0, t.asset_id, t.import_run_id, t.raw_data_hash
        ) for t in transactions
    ]

    try:
        cursor.executemany(sql, data_to_insert)
        conn.commit()
        # The cursor.rowcount tells us how many rows were actually inserted (not ignored).
        print(f"Successfully saved {cursor.rowcount} new transactions to the database.")
    except sqlite3.Error as e:
        print(f"Database error: {e}")
        conn.rollback()
    finally:
        conn.close()

async def import_transactions_csv(account_id: str = Form(...), file: UploadFile = File(...)):
    """
    Accepts a CSV file, parses it, and stores the transactions.
    (PRS Section 5.2)
    """
    # Using the filename extension is more robust than checking the content_type header,
    # which can be inconsistent between different clients (like curl).
    if not file.filename or not file.filename.lower().endswith('.csv'):
        raise HTTPException(status_code=400, detail="Invalid file type. Please upload a CSV.")

    try:
        transactions = csv_importer.parse_standard_csv(contents, account_id)

        save_transactions(transactions)
        
        print(f"Successfully processed {len(transactions)} transactions for account {account_id}.")
        return {
            "message": f"Successfully imported and saved {len(transactions)} transactions.",
            "filename": file.filename,
            "transaction_count": len(transactions)
        }
    except Exception as e:
        # It's good practice to log the actual error for diagnostics
        print(f"ERROR processing file {file.filename}: {e}")
        raise HTTPException(status_code=500, detail=f"Failed to process CSV file: {e}")

import csv
import io
from datetime import datetime
from decimal import Decimal
import hashlib

from ..data_model import Transaction

def parse_standard_csv(file_contents: bytes, account_id: str) -> list[Transaction]:
    """
    Parses a standard CSV file with Date,Description,Amount columns.
    (PRS Section 5)
    """
    transactions = []
    file_stream = io.StringIO(file_contents.decode('utf-8'))
    reader = csv.DictReader(file_stream)

    for row in reader:
        # Basic data validation and cleaning
        try:
            transaction_date = datetime.strptime(row['Date'], '%Y-%m-%d').date()
            amount = Decimal(row['Amount'])
            description = row['Description']
        except (KeyError, ValueError) as e:
            print(f"Skipping row due to parsing error: {row} - {e}")
            continue

        # Create a unique ID for the transaction (PRS Section 8 - dedupe)
        # A simple hash of key fields is a good start.
        raw_data = f"{transaction_date}-{description}-{amount}-{account_id}"
        transaction_hash = hashlib.sha256(raw_data.encode('utf-8')).hexdigest()

        transactions.append(
            Transaction(
                transaction_id=transaction_hash,
                account_id=account_id,
                transaction_date=transaction_date,
                amount=amount,
                description=description,
                raw_data_hash=transaction_hash # Store for deduplication
            )
        )

    return transactions
